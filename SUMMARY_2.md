# Progressive Agent -- Architecture Deep Dive

This document explains how Progressive Agent works internally: the agent loop, tool and skill systems, memory architecture, LLM providers, and background monitors.

## High-Level Architecture

```
Telegram (aiogram 3)
        |
        v
  asyncio.Queue  (unified message pipeline)
        |
        v
     Agent  (orchestrator)
     /    |    \
Router  Memory  Soul
    |
    v
 Dispatcher (Native tool_use / XML fallback)
    |
    v
 LLM Provider (FallbackProvider with multiple backends)
    |
    v
 Tool Executor (50+ tools)
    |
    v
 Response --> Telegram (streamed via edit_message)
```

All incoming messages from Telegram are pushed into a single `asyncio.Queue`. The Agent consumes messages one at a time, assembles context (memories, soul files, skill instructions), calls the LLM, processes tool calls in a loop, and streams the response back to the user via Telegram's `edit_message` API for real-time draft updates.

## The Agent Loop

1. **Message arrives** -- the Telegram channel pushes an `IncomingMessage` into the queue.
2. **Context assembly** -- the Agent loads relevant memories (hybrid search), the soul system prompt (SOUL.md + OWNER.md + RULES.md + traits), and the appropriate skill instructions.
3. **Router** -- selects the best skill based on keyword matching and LLM classification. Skills determine which tools are available for this request.
4. **LLM call** -- the Dispatcher sends the assembled messages and tool definitions to the LLM provider. Streaming is used for real-time output.
5. **Tool execution** -- if the LLM returns `tool_use` blocks, the Agent executes each tool and feeds the results back to the LLM. This loop continues until the LLM produces a final text response.
6. **Response delivery** -- the text is streamed to Telegram via progressive `edit_message` calls (draft updates).
7. **Memory storage** -- the conversation turn is saved to SQLite, and fact extraction runs periodically to capture important information.

## Tool System

Tools are Python classes that implement the `Tool` protocol:

```python
class Tool(Protocol):
    name: str
    description: str
    def definition(self) -> ToolDefinition: ...
    async def execute(self, **kwargs) -> ToolResult: ...
```

Each tool exposes a `definition` property that returns a JSON schema compatible with the LLM's function-calling API. The `execute` method performs the actual work and returns a `ToolResult` with success/failure status and data.

Tools are organized by domain in `src/tools/`. Examples: `search_tool.py` (web search), `email_tool.py` (Gmail), `file_tool.py` (filesystem), `cli_tool.py` (shell commands), `coingecko_tool.py` (crypto prices).

A set of always-available tools (file operations, CLI, git, agent control) is accessible regardless of which skill is active.

## Skill System

Skills are Markdown files (`skills/*/SKILL.md`) with YAML frontmatter, not Python code. They provide instructions that are injected into the LLM's system prompt to guide behavior for specific domains.

```yaml
---
name: web_search
description: Search the internet for information
tools:
  - web_search
  - web_read
  - web_extract
trigger_keywords:
  - search
  - find
  - what is
---

# Instructions

You are an expert at finding information online...
```

The Router selects the matching skill based on trigger keywords and LLM classification. The skill's `tools` list determines which tools are made available for the request. Skills are automatically loaded from the `skills/` directory and support hot-reload.

## Monitor System

Monitors are background jobs managed by APScheduler that run on fixed intervals and push notifications to Telegram without any user prompt. Each monitor:

- Runs on its own schedule (e.g., every 2 minutes, every 4 hours, daily at 08:00).
- Maintains persistent state in JSON files under `data/` to track what has already been reported.
- Sends notifications directly to the owner's Telegram chat.

Current monitors include: CryptoMonitor (price alerts), EmailMonitor (new emails), SubscriptionMonitor (renewal reminders), NewsRadarMonitor (crypto and AI news digest), TwitchMonitor (stream start alerts), YouTubeMonitor (new video alerts), MorningBriefingMonitor (daily digest), and others.

## Memory Architecture

Memory is stored in a single SQLite database (`data/memory.db`) with two indexing strategies:

- **Vector search** -- embeddings generated by OpenAI's `text-embedding-3-small` model, stored and queried via `sqlite-vec` for cosine similarity search.
- **Full-text search** -- SQLite FTS5 index with BM25 scoring for keyword-based retrieval.

The hybrid search pipeline:

1. Run vector similarity search (top candidates by cosine distance).
2. Run FTS5 keyword search (top candidates by BM25 score).
3. Apply temporal decay: `score *= exp(-lambda * days_since_last_access)`.
4. Apply importance boost: `score *= importance * 1.5`.
5. Merge, deduplicate, and return top-K results.

Memory types include `conversation`, `fact`, `preference`, and `task`. Fact extraction runs periodically (every 3 conversation exchanges) to pull out important information from the conversation and store it as discrete, searchable facts.

After a restart, the agent restores recent conversation history from the database, preserving the full tool-call chain so context is not lost.

## LLM Providers

The agent uses a `FallbackProvider` that chains multiple LLM backends in priority order. If the primary provider fails (rate limit, timeout, error), the next one is tried automatically.

```
1. Claude (proxy)         -- primary, via local CLI proxy
2. Gemini 2.0 Flash       -- free tier, 15 RPM, multimodal
3. Mistral large-latest    -- free tier, 1 RPS
4. Cloudflare Workers AI   -- free tier, 10K requests/day
5. OpenAI GPT              -- paid, fast, reliable
6. Claude API              -- paid, emergency fallback
```

Each provider implements the `LLMProvider` protocol with `complete()` and `stream()` methods. All providers are wrapped in a `ReliableProvider` that handles retries, exponential backoff, and rate-limit headers.

The `Dispatcher` abstracts the difference between models that support native function calling (`tool_use` blocks) and those that need XML-based tool instructions as a fallback.

## Configuration

- **`config/agent.toml`** -- main settings: model name, token limits, memory parameters, cost limits, allowed Telegram users.
- **`.env`** -- secrets: API keys and tokens (never committed to version control).
- **`soul/*.md`** -- personality files loaded into the system prompt: identity, owner profile, behavioral rules, personality traits.

## Security

- **Deny-by-default** -- only whitelisted Telegram user IDs can interact with the bot. Unknown users receive troll responses with no tool or memory access.
- **Rate limiting** -- unknown users are limited to 5 messages per 10 minutes.
- **CLI security** -- the shell execution tool uses an executable extraction heuristic with 20+ regex patterns to block dangerous commands and obfuscation attempts.
- **File safety** -- file copy operations validate source paths to prevent exfiltration of sensitive files (.ssh, .env, system directories).
- **Cost tracking** -- daily and monthly spending limits with automatic warnings at configurable thresholds.
